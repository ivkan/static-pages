1. Как работает формула — пошагово

Разберём на одной реальной поездке из истории. После этого вы сможете пройти расчёт самостоятельно для любой другой.

Поездка #1389426 — Dallas, 12,8 миль, transfer

МаршрутDel Norte Ln (Dallas) → Ritz-Carlton Las Colinas
МетроплексDallas, TX (tier 2)
Billingtransfer (не hourly)
Дистанция12,8 миль
Аэропортнет
Реальный TotalCost (что получил vendor)$71,40

Калиброванные параметры Tier 2: base_fare=$90,20, per_mile_rate=$1,731, min_fare=$60. Dallas rate_adjustment=0,61. Margin target = 40%.

1Базовая стоимость

baseCost = base_fare + per_mile_rate × distance
= $90,20 + $1,731 × 12,8 = $90,20 + $22,16 = $112,36

Это «сырая» стоимость поездки для vendor, до всех корректировок. Сколько машина должна зарабатывать за фиксированный выезд + пройденные мили.

Если поездка airport: к base_fare добавляется фиксированная надбавка из tier — airport_pickup_surcharge (если аэропорт на выезде) или airport_dropoff_surcharge (если аэропорт на приезде). Для Tier 2: pickup = $15,00, dropoff = $7,50. Формула становится: baseCost = base_fare + airport_surcharge + per_mile_rate × distance. В нашем примере поездка не airport, поэтому надбавка не применяется. Airport-надбавки не калибруются — остаются на текущих серверных значениях.

2Deadhead (холостой пробег) + Wait (ожидание)

baseCost += deadhead_cost + wait_charge
Обе точки в пределах main_radius=15 миль от центра Dallas (3,6 и 6,7 миль) → deadhead=0. Ожидание не записано → wait=0.
baseCost = $112,36 (не изменилось)

Если бы vendor ехал издалека за пассажиром — прибавили бы стоимость пустого пробега по ставке tier. Для поездок внутри города это ноль.

3Пол по min_fare

baseCost = max(baseCost, min_fare)
= max($112,36, $60) = $112,36

Защита от слишком коротких поездок, которые не окупают выезд. Для нашего расчёта не сработала.

4Suburban rate factor

if suburban: baseCost × suburban_rate_factor
Обе точки в main_radius (не в «пригородном кольце» 15–25 миль) → множитель не применяется.
baseCost = $112,36

Если бы обе точки оказались в расширенной зоне (за пределами main, но в пределах extended) — применился бы множитель 1,30. Визуально — два кольца вокруг центра города.

5Добавление маржи (наценка клиенту)

price = baseCost / (1 − base_margin_target) = baseCost / 0,6
= $112,36 / 0,6 = $187,27

Делим на 0,6, потому что цель маржи — 40%. Обратная величина: vendor_cost = 60% от customer_price. Значит customer_price = vendor_cost / 0,6. Это базовая бизнес-модель MUV.

6Per-metroplex множитель rate_adjustment

price = price × rate_adjustment
= $187,27 × 0,61 = $114,23

Локальная корректировка Dallas. В Dallas реальные vendor-ставки ниже, чем даёт чистая Tier 2 — поэтому множитель 0,61. В Tijuana, наоборот, 2,118 (см. раздел 5).

7Финальный min_fare

price = max(price, min_fare)
= max($114,23, $60) = $114,23

Это итоговая цена для клиента.

✓Оценка для калибровки (effectiveCost)

effectiveCost = baseCost × rate_adjustment
= $112,36 × 0,61 = $68,54
Сравниваем с реальным TotalCost = $71,40
Ошибка = |68,54 − 71,40| / 71,40 = 4,0% — попадает в ±10% ✓

Это та цифра, по которой мы измеряем точность формулы. Мы сравниваем её не с customer_price (который искажён скидками, контрактами, событиями), а с TotalCost — чистым сигналом того, что vendor реально получил.

2. Как мы измеряем точность

В три шага:

  1. Берём историческую поездку из базы. Знаем реальный TotalCost (что vendor получил).
  2. Прогоняем её через формулу с откалиброванными параметрами — получаем effectiveCost.
  3. Считаем относительное отклонение. Если ≤ 10% — поездка «точная».

Применяем ко всем 15 021 поездкам. Точность = процент точных поездок.

Распределение точности по допускам

ДопускТочностьЧто это значит
±5%43,0%формула почти в точку
±10%62,7%текущий стандарт качества
±15%73,3%
±20%79,8%
±25%84,9%«85%» при этом допуске
±30%88,5%

Медиана ошибки 6,6% — то есть у половины всех поездок формула угадывает точнее, чем ±7%.

3. Откуда берутся новые значения параметров

Калибровка идёт в две фазы. В каждой итерации Фазы A последовательно пересчитываются несколько групп параметров, после чего измеряется точность — и цикл повторяется, пока изменения не перестанут улучшать результат. Фаза B включается после сходимости Фазы A.

Что калибруется

Параметр Scope Источник значения Где настраивается
base_fare на tier (5 штук) линейная регрессия: пересечение линии «cost vs distance» по всем поездкам tier Фаза A, шаг 1
per_mile_rate на tier (5 штук) линейная регрессия: наклон той же линии Фаза A, шаг 1
sedan_hourly_rate на tier (5 штук) Отдельная регрессия по hourly-поездкам (где billing_type = hourly) Фаза A, шаг 2
rate_adjustment на метроплекс (148 изменений) median(TotalCost / predicted_baseCost) по поездкам метроплекса Фаза A, шаг 3
tier метроплекса на метроплекс (35 переназначений) Перебор всех 5 tier'ов, выбирается тот, который даёт больше всего попаданий в ±10% Фаза B

Не калибруются (оставлены на старых значениях): min_fare, deadhead_rate_per_mi, airport_surcharge, радиусы зон и suburban_rate_factor. Радиусы меняются только на стороне сервера (Stefan и Саша это уже сделали до калибровки).

Примечание: в API и БД MUV эти же поля записаны с суффиксом _usd (например, base_fare_usd) — суффикс обозначает валюту. В этом документе для краткости суффикс опущен.

Фаза A — итеративная подгонка параметров

Внутренний цикл, до 100 итераций. На каждой итерации — четыре шага в фиксированном порядке:

1Линейная регрессия для base_fare и per_mile_rate каждого tier

только transfer-поездки · distance > 3 миль · без промежуточных остановок

Для каждого tier берём эти отфильтрованные transfer-поездки и строим линейную регрессию TotalCost = base_fare + per_mile_rate × distance. Подбираются такие два коэффициента, при которых формульный cost в среднем меньше всего отличается от исторического TotalCost по всем поездкам этого tier.

2Регрессия для sedan_hourly_rate каждого tier

только hourly-поездки

Отдельная регрессия по формуле TotalCost = sedan_hourly_rate × max(2h, hours). Считается независимо от transfer, потому что это другая бизнес-модель: здесь дистанция не учитывается, цена зависит от часов.

3Пересчёт rate_adjustment для всех 148 метроплексов

все поездки метроплекса (transfer + hourly)

Для каждого метроплекса: прогоняем все его поездки через формулу с обновлёнными tier-параметрами (формула сама выбирает правильную ветку — transfer или hourly). Получаем предсказанный baseCost для каждой поездки. Затем rate_adjustment = median(TotalCost / baseCost) по всем поездкам метроплекса.

Что это значит: для каждой поездки считаем отношение TotalCost / baseCost — во сколько раз реальная стоимость отличается от формульного предсказания. Если отношение > 1 — формула занижает, если < 1 — завышает. Медиана этих отношений по всем поездкам города и становится rate_adjustment — коэффициентом, который подстраивает формулу под локальный уровень цен. Пример: в Tijuana медиана ≈ 2,12 → формула стабильно в 2× занижает → rate_adjustment=2,118.

Медиана — серединное значение упорядоченного списка: половина чисел меньше, половина больше. В отличие от среднего, одна аномальная поездка (например, с ratio=10) не сдвинет её — медиана отражает типичный случай. Пример: для [1,8, 2,0, 2,1, 2,3, 10,5] медиана = 2,1, а среднее = 3,74.

4Оценка точности и решение о продолжении

все 15 021 поездок (transfer + hourly)

Считаем % поездок в ±10% по всем поездкам. Если изменение параметров дало прирост — продолжаем следующую итерацию. Остановка, если: точность ≥ 90% (успех), все дельты параметров < 0,001 (нет изменений), 3 итерации подряд без прироста (застой), либо 100 итераций (safety cap).

Фаза B — переназначение tier'ов метроплексам

все поездки каждого метроплекса

После сходимости Фазы A для каждого метроплекса перебираем все 5 tier'ов и замеряем точность. Если какой-то tier даёт прирост ≥5% попаданий — назначаем его этому метроплексу. Затем снова запускается Фаза A с новыми tier-назначениями. При ухудшении общей точности — откат и блокировка метроплекса.

Откат: перед применением новых tier'ов алгоритм сохраняет копию состояния. Если после переназначения общая точность упала — восстанавливает сохранённую копию, как будто переназначения не было.

Блокировка: каждый метроплекс, который уже был переназначен (успешно или с откатом), добавляется в список «заблокированных» и в следующих раундах пропускается. Это предотвращает бесконечную осцилляцию (например, tier 4 → 3 → 4 → 3 → ...) — каждый метроплекс участвует в Фазе B максимум один раз.

Результат: 35 метроплексов получили другой tier (например, Tier 4 → 3 для переоценённых рынков).

4. Почему rate_adjustment такой разный?

После подгонки tier-параметров остаётся локальный разброс между городами. rate_adjustment — это per-metroplex корректировка, которая описывает систематическое отклонение данного города от «средней температуры по больнице».

Медианная реальная стоимость поездки по городам (взяты по 33–59 поездок каждый)

Tijuana → ×2,118

Медианный TotalCost $165 на коротких поездках. Реальные vendor-ставки систематически в 2× выше, чем даёт чистая Tier 2. Возможные причины: специфика рынка, дефицит vendors, премия за риск.

Dubai → ×0,234

Медианный TotalCost $35 — в 3× ниже, чем предсказала бы Tier 4 (~$110 на 4,8 миль). Локальные тарифы дешевле. Без калибровки клиент платил бы втрое дороже факта.

Большой rate_adjustment — это реакция на систематическое расхождение между формулой и фактами. Чем стабильнее отклонение в городе (одинаково высокое у всех 53 поездок Tijuana, одинаково низкое у всех 33 поездок Dubai), тем увереннее мы можем его зафиксировать как локальный множитель.

6. Главный тезис

Новые цифры выглядят большими только на фоне старых — но старые были просто неверными. Они давали 32% попаданий и систематически расходились с реальностью на ×2–×3 в крупных городах. Каждое новое значение — результат подгонки под реальные TotalCost 15 021 поездки.

Что важно:

  • Мы изменяем customer_price, а не vendor_payout. Vendor-выплата в MUV договорная, не формульная. Калибровка улучшает только то, что видит клиент.
  • 62,7% — структурный потолок этой формулы. Чтобы перейти к 85–90%, нужны новые входные данные (время суток, конкретный vendor, тип маршрута) — это отдельная задача.
  • Текущая калибровка готова к деплою. 2× прирост к старой точности, стабильность проверена на 5 batches review от Stefan.
